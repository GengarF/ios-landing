## 和 C++ 的关键对应关系

### 文件拆分
- Swift 多文件组织 ≈ C++ 中按 `.h` / `.cpp` 或模块拆分
- Swift 没有传统头文件，但同样需要按职责组织代码

### 数据模型
- Swift `struct` 数据模型 ≈ C++ 中清晰设计的 `struct`
- 字段类型要有语义，而不是全部用字符串糊起来

### 属性和方法
- 存储属性 ≈ 成员变量
- 方法 ≈ 成员函数
- 计算属性 ≈ getter 风格的派生结果
- `static` ≈ 静态成员

### protocol
- protocol ≈ 接口 / 抽象基类 / 纯虚函数约束
- protocol extension ≈ 接口默认实现
- protocol 作为参数 ≈ 接收抽象接口引用

### value / reference
- `struct` 值语义 ≈ 普通对象拷贝
- `class` 引用语义 ≈ 多个指针或引用指向同一个对象
- `inout` ≈ C++ 引用参数 `T&`

### 小项目组织
- Model / Store / Helper / Main 的划分 ≈ C++ 项目中模型、管理器、工具函数和入口函数的职责划分

## 这一阶段我觉得最重要的 Swift 特点

### 1. Swift 很重视值类型
`struct` 在 Swift 里非常常用。  
很多模型、数据对象、简单业务结构都可以优先从 `struct` 开始。

### 2. Swift 鼓励清晰的模型表达
一个好的模型不仅要能存数据，还应该让数据结构清楚、有语义。

比如：
- `durationSeconds` 比 `duration` 更明确
- `isPublished: Bool` 比 `"published"` 字符串更可靠
- `Author` 作为模型比 `authorName: String` 更有扩展性

### 3. protocol 是 Swift 代码组织的重要工具
protocol 让不同类型可以共享一套能力，而不是必须通过继承建立关系。

这和 Swift 的风格非常契合。

### 4. 小项目需要职责划分
当代码从单题进入小项目后，最重要的不只是“能跑”，还要考虑：
- 哪些是数据模型
- 哪些是管理逻辑
- 哪些是工具函数
- 哪些是入口流程

### 5. AI review 对小项目很有帮助
AI 在这个阶段最适合做：
- 结构解释
- 命名 review
- 文件职责 review
- 小步优化建议

但不适合无脑让它大规模重构。
