# Bug 排查工作流模板

这个模板的目标不是把每个 Bug 都写成长文，而是帮助自己在陌生项目里稳定推进：

- 快速复现
- 快速缩小范围
- 用证据验证假设
- 小步修复
- 做到可解释、可回归、可交付

适用场景：

- mentor / leader 分配了一个线上或测试环境 Bug
- 自己写项目时遇到功能异常、崩溃、编译失败、状态不同步
- 面对不熟悉的 iOS 工程，需要尽快定位问题
- 结合 Codex、Claude 等 AI 工具，提高阅读代码、生成假设、补测试和写 PR 的效率

## 6. AI 协同排查方式

AI 的作用不是替你负责，而是帮你更快完成：

- 读代码
- 找入口
- 总结链路
- 生成假设
- 设计验证方式
- 起草修复方案
- 补测试和补 PR 描述

最终判断仍然要由自己完成。你需要能解释：

- Bug 是怎么复现的
- 根因为什么成立
- AI 改了什么
- 为什么这样改
- 验证覆盖了哪些路径

### 6.1 什么时候适合用 AI

适合：

- 陌生项目不知道入口在哪里
- crash 栈很长，需要快速定位业务调用
- 不确定某段 Swift / UIKit 代码的意图
- 想列出可能根因和验证顺序
- 想让 AI 帮你补单元测试、mock 数据、边界 case
- 修复后让 AI 做 code review，找回归风险
- 写 PR 描述、排查记录、mentor 汇报

不适合直接依赖：

- 没有复现信息就让 AI 猜
- 没看 diff 就接受 AI 的大改
- 让 AI 一次性重构大片代码
- 让 AI 编造线上原因、接口协议或业务规则
- 对支付、登录、安全、隐私、数据删除等高风险路径直接自动改

## 14. 校招生使用建议

### 14.1 日常真实使用时不要全填

小问题只填：

- 现象
- 复现步骤
- 根因
- 修复
- 验证

复杂问题再补：

- 影响范围
- 排查记录
- 风险
- 复盘

如果使用了 AI，小问题额外补一句：

- AI 帮了什么：
- 我验证了什么：

### 14.2 你最需要练的不是“会不会改”，而是“能不能稳定定位”

校招生进入大厂后，常见困难不是 Swift 语法完全不会，而是：

- 不熟悉项目结构
- 不知道从哪里下断点
- 分不清现象和根因
- 一次改太多导致无法判断哪个改动生效
- 修完当前路径但漏掉回归路径
- 不会向 mentor 汇报排查进展

所以每次排 Bug 都强迫自己回答四个问题：

1. 这个问题是否可复现？
2. 我现在怀疑什么？
3. 我用什么证据证明它成立或不成立？
4. 修完后我验证了哪些路径？

使用 AI 后，再加两个问题：

1. AI 的结论有没有证据？
2. 我能不能脱离 AI 向 mentor 讲清楚？

### 14.3 陌生项目里优先建立三张图

- 页面图：用户从哪里进来，经过哪些页面
- 数据图：数据从接口 / 缓存到 UI 的链路
- 状态图：loading、empty、error、success、refresh、login 等状态如何切换

不用画得漂亮，能让自己说清楚就够。

AI 可以帮你把三张图整理得更快，但第一版最好自己先画。这样你不会被 AI 的错误链路带偏。

### 14.4 和 mentor 沟通的节奏

不要卡很久才问，也不要没做任何排查就问。

比较好的节奏：

- 先自己复现
- 找到最小链路
- 提出 2-3 个怀疑点
- 验证其中 1-2 个
- 如果仍然卡住，带着记录去问

提问格式：

```text
我现在能稳定复现：xxx。
链路大概是：A -> B -> C。
我验证了：假设 1 不成立，因为 xxx；假设 2 可能成立，因为 xxx。
我现在卡在：不知道 xxx 的状态是在哪里被覆盖的。
想请你帮我确认一下排查方向是否对。
```

如果 AI 参与了排查，可以补一句：

```text
我也让 AI 帮我读了相关链路，它给出的方向是 xxx。
我已经验证了其中 xxx 成立 / 不成立。
```

## 15. 这套流程是否足够

结论：对校招生和入职初期 iOS 开发来说，这套流程已经足够覆盖绝大多数日常 Bug。

它能覆盖：

- 编译失败
- 崩溃
- UI 展示异常
- 数据不一致
- 网络接口异常
- 异步回调和线程问题
- 生命周期和内存问题
- 陌生项目快速定位
- 修复后的回归验证和汇报
- AI 协同读代码、生成假设、辅助修复、辅助 review

它还不能完全覆盖：

- 大型性能问题
- 复杂线上事故
- 多端协同问题
- 底层系统问题
- 复杂架构迁移引起的问题
- 需要大量监控、灰度、日志平台支持的问题

结合 AI 后，它会更高效，尤其是在陌生项目阅读、调用链整理、边界 case 枚举、测试补充、PR 表达这些环节。但效率提升的前提是：你把 AI 当作协作工具，而不是责任主体。

当前最重要的是把这个流程练熟：每个 Bug 都能做到“复现清楚、假设清楚、证据清楚、修复清楚、验证清楚”。如果使用 AI，还要做到“AI 帮助清楚、人工验证清楚、自己能讲清楚”。
